HDC 从零开始 🧠

超维计算(Hyperdimensional Computing)。给纯小白:每个 蓝词 点一下看人话。下面的向量能点能算。

HDC 是我们这块芯片上跑的另一种"大脑算法"——不靠神经网络,靠几万维的随机向量做"绑定/打包/比对"。它天生 1-bit、抗噪、一次就学会、永不遗忘——正好长在 ReRAM 这种又省电又不太精确的硬件上。这页讲清它是什么,并把我们所有 HDC 的代码仓库挂在一起。

① 一个想法 = 一串一万维的随机数

HDC 把每个概念(比如"猫""红色""国王")都表示成一条很长很长(1万~10万维)的向量,每一维只是 +1 或 −1。一开始就是随机给的。

概念 A(猫)
概念 B(狗)
相似度 A·B

⭐ 神奇之处:任意两个随机长向量,相似度都接近 0(几乎垂直)。维度越高越是如此。这意味着空间大到几乎装得下无穷多个互不干扰的概念——这是 HDC 一切的根。为什么?

② 三个动作,就能搭出"思考"

HDC 只有 三种运算,但够用:

🔗 BIND 绑定(把两个东西"扣"成一对)

把 A 和 B 逐位相乘 → 得到一个和谁都不像的新向量 C。用来表示"A 配 B"这种键值对(比如"颜色=红")。而且可逆:拿 C 再和 B 绑一次,就把 A 原样还原出来。

结果 C
C 和 A 像吗

点"绑定"看看:C 和 A、和 B 都不像(相似度≈0)——它是个全新的东西。但信息没丢,解绑能完整拿回来。

📦 BUNDLE 打包(把一堆东西塞进"一个记忆")

把几个向量加起来取符号 → 得到一个和每个成员都有点像的向量 M。用来表示集合 / 一段记忆("我见过 X、Y、Z")。之后拿任意东西去比,就知道它在不在这个记忆里。

记忆 M
成员 X 在里面?
陌生 R 在里面?

点"打包":成员 X 和 M 明显相似(在记忆里),陌生向量 R 和 M ≈0(不在)。这就是 HDC 的"记得"。

↪ PERMUTE 移位(记住"顺序")

把向量整体平移几位 → 用来编码先后/位置("先 A 后 B" ≠ "先 B 后 A")。这样连句子、时间序列都能装进一条向量。

③ 为什么 HDC 和我们的 ReRAM 芯片是绝配?

HDC 的天性普通神经网络为什么配 ReRAM
1-bit(只要 +1/−1)要 8~16 bit 浮点ReRAM cell 本就难存高精度,HDC 只要 1 bit,正好
抗噪(30% 翻转还能用)少量噪声就崩ReRAM 电阻会飘,HDC 不在乎
一次就学会、永不遗忘要几百步梯度、会灾难性遗忘写一次就记住,契合非易失存储
追加是 O(1)(随手加)重训或建索引存内直接 bundle 进去,不用搬
一句话:神经网络要"精确"——而 ReRAM 给不起精确;HDC 要的恰恰是"高维 + 1-bit + 抗噪",这正是 ReRAM 便宜又省电能给的。算法的脾气和硬件的脾气对上了。详见 ReRAM 101 §⑤ 多电平。

④ HDC 资源中心(我们所有 HDC 代码挂这)

这一节是个会长大的中心——现在挂着这些,以后的 HDC 项目都往这加:

🧱 hdc-ops
底层武器库:HDC 代数核心(bind/bundle/permute/unbind + 编码/结构/查询/诊断/分页存储),纯 numpy + 可选 MLX 后端。双语 docstring,每个函数带"何时用"。pip install -e . 就能跑。MIT。
⚡ hdc-neon
用 NEON SIMD 加速 HDC 检索,比 numpy 快 ~89×。证明 HDC 在普通 CPU 上就已经飞快,搬到 ReRAM-CIM 会更夸张。Apache-2.0。
🔧 torchhd(fork · reram-cim-backend)
在主流 HDC 库 torchhd 上加了一个可直接替换的 ReRAM-CIM 后端 —— 2 行代码,把标准 HDC 切到我们的芯片仿真上跑。

这里的“随机”就是抛硬币:每一维独立地 50% 给 +1、50% 给 −1。一万维就抛一万次。正因为是独立随机,两个这样生成的向量几乎不可能碰巧很像。
在很高的维度里,两个随机 ±1 向量“同号的位”和“异号的位”几乎各占一半,点积≈0,所以相似度≈0(近乎垂直/正交)。维度越高,这个“≈0”越稳。好处:你可以塞进海量互不干扰的概念,还能用“相似度”可靠地区分它们——这就是“超维”的超能力。
HDC 只有三种运算:BIND 绑定(逐位相乘,把两个扣成一对、且可逆)、BUNDLE 打包(相加取符号,把多个并成一个“集合/记忆”、和每个成员都像)、PERMUTE 移位(整体平移,编码顺序/位置)。就这三招,能搭出键值对、集合、序列——足够表达结构化的“想法”。
